需求分析 → 框架搭建 → 原型设计 → 后端实现 → 调测验证

01
DISCUSS
需求讨论与可行性分析
Feasibility

向 Trae 提出想法："想给 PDF 工具接入本地多模态模型，可行吗？" Trae 评估技术可行性、对比多方案，并明确需求范围——确定项目可做。

对比 BLIP-base / BLIP-large / GIT / OFA 多个模型方案
选定 BLIP-base：约 1GB，CPU 推理 2-5 秒/图
明确需求范围：提取 → AI 描述 → 审校 → 写入 /Alt 完整闭环
02
SCAFFOLD
搭框架，后端核心工序跑通
Foundation

项目骨架搭建后，优先把 PDF 处理的核心工序跑通，先不接 AI，确保基础链路稳定，为后续接入留好接口。

PyMuPDF 提取 → pHash 去重 → 过滤装饰图 → pikepdf 写入
提取→过滤→写入 基础链路稳定可靠
通过依赖注入预留 ai_captioner 接口
03
PROTOTYPE
前端 HTML 搭建（原型图逻辑）
Prototype

这一步类似产品经理绘制原型图——先用 HTML+CSS 把页面结构和交互流程搭出来，确定"用户看到什么、怎么操作"，再倒推后端需要提供哪些接口。

⬆️
上传区
📄
PDF 缩略图
🖼️
图片列表
✏️
审校面板
📊
进度条
💬
Toast 提示
04
IMPLEMENT
根据前端实现后端代码
API Driven

前端搭好后，后端 API 设计就有了明确依据。Trae 根据前端调用的字段，反向实现后端逻辑，前后端字段一一对应。

POST /api/upload · 文件上传与任务创建
POST /api/process · 触发处理，返回 AI 状态
GET /api/results/:id · 拉取结果与图片预览
状态字段对齐：ai_model_available / used_fallback
05
DEBUG
调测与迭代
Verify

端到端联调中发现并修复多个典型问题，最终在云端验证通过：

模型加载失败 · symlink 机制导致 local_files_only 失效 → 新增 _find_model_path()
本地无缓存 → 新增 BLIP_MODEL_PATH 环境变量，提供本地配置指南
前端无错误提示 → 添加失败判断，模型不可用时显示 6 秒错误 toast
云端验证："ai_model_available": true
"used_fallback": false · 端到端链路完整跑通

端到端链路打通
